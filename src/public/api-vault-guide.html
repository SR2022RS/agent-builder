API Vault Build Instructions

How to securely store, manage, and rotate API keys for your AI agent stack

Why This Matters

API keys are the single point of failure in every AI agent setup. One leaked key can lead to unauthorized usage, unexpected bills, or full account compromise. This guide covers three approaches — pick the one that fits your workflow.

Option A

.env File (Local Development)

Examples: dotenv, direnv, docker-compose env_file

Best for: Solo devs, local testing
Effort: Low Security: Basic

Setup Steps

1

Create your .env file

# .env — NEVER commit this file
OPENAI_API_KEY=sk-proj-xxxxxxxxxxxxxxxx
ANTHROPIC_API_KEY=sk-ant-xxxxxxxxxxxxxxxx
REPLICATE_API_TOKEN=r8_xxxxxxxxxxxxxxxx
2

Add .env to .gitignore

# .gitignore
.env
.env.local
.env.*.local
3

Load in your app

# Python
from dotenv import load_dotenv
load_dotenv()

# Node.js
import 'dotenv/config'
4

Create a .env.example template

# .env.example — commit this file
OPENAI_API_KEY=your-key-here
ANTHROPIC_API_KEY=your-key-here
REPLICATE_API_TOKEN=your-key-here
Pro Tip
  • Use direnv to auto-load .env when you cd into the project
  • Run git secrets --install to prevent accidental key commits
Option B

Platform Secrets (CI/CD & Hosting)

Examples: Railway Variables, Vercel Env, GitHub Secrets, AWS SSM

Best for: Deployed apps, CI pipelines
Effort: Low-Med Security: Good

Setup Steps

1

Add secrets in your platform dashboard

Navigate to your project settings → Environment Variables and add each key.

2

Use named env vars per project

# Good — scoped and clear
MYAPP_OPENAI_KEY=sk-proj-xxx
MYAPP_ANTHROPIC_KEY=sk-ant-xxx

# Bad — generic, easy to leak across projects
API_KEY=sk-xxx
3

Scope secrets per environment

Use separate values for development, preview, and production environments. Never share keys across stages.

4

Access in code via process.env

# The platform injects these at runtime
const key = process.env.MYAPP_OPENAI_KEY
Platform-Specific Notes
  • Railway: Variables are encrypted at rest and injected at deploy time
  • Vercel: Use "Sensitive" toggle to mask values in logs
  • GitHub Actions: Use secrets.YOUR_KEY in workflow YAML
Option C RECOMMENDED

Secret Vault Service

Examples: 1Password Secrets, Infisical, Doppler

Best for: Teams, client deployments, production
Effort: Medium Security: Highest

Why Use a Vault?

  • Centralized control — one place to manage all secrets across every project and environment
  • Automatic rotation — schedule key rotation without redeploying
  • Audit trail — full log of who accessed which secret and when
  • Access control — grant per-project, per-team, or per-environment permissions
  • Zero plaintext — secrets are never stored in files, env vars, or git history

Setup Steps

1

Choose your vault provider

Free Tier
1Password

Best for teams already using 1Password

Open Source
Infisical

Self-hostable, great for full control

Easiest DX
Doppler

Best CLI & integrations out of the box

2

Create a project and environments

Set up separate environments (dev, staging, production) so each stage pulls its own keys.

# Doppler example
doppler projects create my-ai-agent
doppler environments create --project my-ai-agent dev
doppler environments create --project my-ai-agent production

# Infisical example
infisical init
# Follow prompts to select org → project → environment
3

Add your secrets to the vault

# Doppler
doppler secrets set OPENAI_API_KEY=sk-proj-xxx --project my-ai-agent --config production
doppler secrets set ANTHROPIC_API_KEY=sk-ant-xxx --project my-ai-agent --config production

# Infisical
infisical secrets set OPENAI_API_KEY=sk-proj-xxx --env=prod
infisical secrets set ANTHROPIC_API_KEY=sk-ant-xxx --env=prod

# 1Password CLI
op item create --category=api_credential \
  --title="OpenAI Production" \
  --vault="AI Agent Keys" \
  "credential=sk-proj-xxx"
4

Inject secrets at runtime (no .env needed)

# Doppler — wraps your process and injects env vars
doppler run --project my-ai-agent --config production -- node server.js

# Infisical — same concept
infisical run --env=prod -- node server.js

# 1Password — uses secret references
op run --env-file=.env.tpl -- node server.js

# .env.tpl (1Password template)
OPENAI_API_KEY=op://AI Agent Keys/OpenAI Production/credential
5

Connect to your deployment platform

All three providers offer native integrations with Railway, Vercel, AWS, and GitHub Actions.

# Doppler → Railway integration
# 1. Go to Doppler dashboard → Integrations → Railway
# 2. Authorize and select project + environment
# 3. Secrets auto-sync on every deploy

# Infisical → GitHub Actions example
# Uses official GitHub Action to inject secrets
- uses: Infisical/secrets-action@v1
  with:
    env: production
    projectId: ${{ secrets.INFISICAL_PROJECT_ID }}
6

Set up automatic key rotation

Configure rotation schedules so keys are cycled without manual intervention.

# Doppler: rotation is configured per-secret in dashboard
# Settings → Secret → Enable Rotation → Set interval (e.g., 90 days)

# Infisical: use the rotation API
infisical secrets rotation create \
  --secret-name OPENAI_API_KEY \
  --interval 90d \
  --env=prod

# 1Password: set expiry reminders, rotate manually or via Connect API
7

Grant team access with least privilege

Add team members with scoped roles — developers see dev secrets, only ops sees production.

# Doppler
doppler team members add dev@team.com --role=viewer --project my-ai-agent --config dev
doppler team members add ops@team.com --role=admin --project my-ai-agent --config production

# Infisical
# Dashboard → Members → Add → Assign role per environment
Quick Start Recommendation
  • Solo + simple? Start with Doppler — best CLI, 5-minute setup, generous free tier
  • Team + compliance? Use Infisical — open source, self-hostable, full audit logs
  • Already use 1Password? Add the Secrets Automation add-on — no new tool to learn
How It Works in Practice

Your code stays identical to Option A or B — it still reads process.env.OPENAI_API_KEY. The vault CLI wraps your process and injects secrets as environment variables at startup. No code changes needed. The difference is that secrets live in an encrypted vault with access controls, rotation, and audit logs — not in a flat file or platform dashboard.

Best Practices (All Options)

Always Do This
  • Rotate all keys every 90 days minimum
  • Use named env vars per project, not generic names
  • Never log API keys in console output
  • Monitor usage dashboards weekly for anomalies
  • Run scan script after any new agent or project setup
  • Test agents with new keys before revoking old ones
Never Do This
  • Hardcode any key in source code files
  • Store keys in Notion, Slack, or any chat tool
  • Share keys in screenshots (blur that section)
  • Use the same key across production and development
  • Ignore rate limit errors thinking they are just bugs
